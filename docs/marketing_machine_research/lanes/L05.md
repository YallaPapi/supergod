# L05 - Conversion Stack (Research)

## 1) Three In-Lane Options

### Option A (Lean + Fast): Carrd landing + Airtable CRM + Zapier automation + Calendly + Stripe Payment Links

**Why this option:** fastest low-cost path to launch a conversion stack in 1-2 days, with enough automation for first 10 clients.

**Landing stack**
- `Carrd` one-page offer site with clear CTA blocks: `Book Call` and `Buy Starter Package`.
- Embedded lead form (Carrd native form or embed) and trust proof section.

**CRM schema (Airtable base)**
- `Leads` table fields: `lead_id`, `created_at`, `name`, `email`, `phone`, `source_channel` (Reddit/IG/Referral/Other), `offer_interest`, `intent_score` (0-100), `status` (New/Qualified/Booked/Proposal/Sold/Lost), `owner`, `next_action_at`, `last_touch_at`, `notes`.
- `Activities` table: `activity_id`, `lead_id`, `type` (email/call/dm/form), `timestamp`, `outcome`, `next_step`.
- `Deals` table: `deal_id`, `lead_id`, `package`, `price`, `payment_link`, `payment_status`, `close_date`, `reason_lost`.

**Automation handoff points**
- Form submit -> Zapier -> create/update Airtable lead + assign owner + send Slack/Email alert.
- Qualified lead -> Calendly booking link by intent branch.
- Closed-won -> Stripe Payment Link completed -> Airtable deal update -> onboarding task creation for content delivery workflow.

**Evidence notes**
- Carrd Pro supports custom domains, forms, embeds, and payment-enabled form flows.
- Zapier Free/Pro supports no-code automation and webhooks for multi-step handoffs.
- Airtable automations support triggers/actions (including form submission/webhook triggers).
- Stripe Payment Links are no-code and shareable across web/social/email.

**Sources (with evidence date)**
- Carrd Pro plans/features: https://carrd.co/pro (accessed 2026-03-03)
- Zapier pricing/features: https://zapier.com/pricing (accessed 2026-03-03)
- Airtable automations: https://support.airtable.com/docs/es/getting-started-with-airtable-automations (published ~2022-06, accessed 2026-03-03)
- Stripe Payment Links docs: https://docs.stripe.com/payment-links (accessed 2026-03-03)

### Option B (CRM-First): HubSpot Free CRM + HubSpot Forms + HubSpot Meetings + Stripe Payment Links

**Why this option:** best structure/reporting early, fewer integration edges, cleaner pipeline management.

**Landing stack**
- Fast landing page (can be external site) with embedded HubSpot form.
- CTA flow: `Submit brief` -> `Book discovery` -> `Pay deposit`.

**CRM schema (HubSpot properties)**
- Contact properties: `Lead Source`, `Content Goal`, `Monthly Budget`, `Platform Priority`, `Offer Type`, `Lifecycle Stage`, `Lead Status`, `Owner`, `Next Activity Date`.
- Deal properties: `Pipeline`, `Deal Stage`, `Package`, `MRR/Project Value`, `Close Date`, `Lost Reason`.

**Automation handoff points**
- HubSpot form submission auto-creates contact and routes into CRM.
- Meeting scheduler syncs booking activity to contact timeline.
- Payment Link purchase event updates deal stage (needs workflow/integration depending on payment method).

**Evidence notes**
- HubSpot Free CRM provides core CRM capability and free tier availability.
- HubSpot Forms captures leads into CRM directly.
- HubSpot Meeting Scheduler supports booking with CRM-connected scheduling.

### Option C (All-in-One Agency Stack): HighLevel (landing/funnel + CRM pipeline + workflows)

**Why this option:** strongest native “single system” for agency-style lead->deal->automation flow, but higher complexity/cost.

**Landing stack**
- HighLevel funnel/landing pages with native forms and booking widgets.
- CTA flow with pipeline stage movement handled in-platform.

**CRM schema (HighLevel opportunities + custom fields)**
- Contact fields: `source_channel`, `service_interest`, `budget_band`, `urgency`, `platform_mix`, `owner`.
- Opportunity/pipeline fields: `stage`, `offer_value`, `close_probability`, `last_contacted`, `next_step`, `loss_reason`.

**Automation handoff points**
- Form submit -> create/update contact/opportunity.
- Pipeline stage changes trigger workflow branches (follow-up, reminders, proposal, onboarding).
- Won stage triggers client onboarding and production kickoff tasks.

**Evidence notes**
- HighLevel lists Starter at $97/mo and Unlimited at $297/mo on official pages.
- HighLevel support docs confirm pipelines as core journey mapping and workflows as native automation layer.

## 2) Lane Winner With Explicit Score Math (0-100)

### Scoring model
- Speed to launch (weight 30)
- First-10-clients cost efficiency (weight 20)
- Automation/handoff depth (weight 25)
- CRM clarity/reporting (weight 15)
- Flexibility / lock-in risk (weight 10)

Score formula: `Total = sum((criterion_score_0_to_10 / 10) * criterion_weight)`

### Score table
- Option A: `((9/10)*30) + ((9/10)*20) + ((7/10)*25) + ((6/10)*15) + ((8/10)*10) = 27 + 18 + 17.5 + 9 + 8 = 79.5`
- Option B: `((8/10)*30) + ((8/10)*20) + ((8/10)*25) + ((8/10)*15) + ((7/10)*10) = 24 + 16 + 20 + 12 + 7 = 79.0`
- Option C: `((7/10)*30) + ((6/10)*20) + ((9/10)*25) + ((8/10)*15) + ((5/10)*10) = 21 + 12 + 22.5 + 12 + 5 = 72.5`

### Winner
**Option A wins (79.5/100)** for fastest practical execution to first 10 paying clients with low upfront cost and acceptable automation.

**UNPROVEN:** Option A will outperform Option B on close-rate after 30+ leads. (Needs live funnel data.)

## 3) Key Failure Modes + Mitigations

- Failure mode: lead leakage between form -> CRM -> booking.
- Mitigation: add daily reconciliation view (`new leads without owner`, `qualified leads without meeting`, `won without payment`).

- Failure mode: slow speed-to-lead reduces conversion.
- Mitigation: instant owner alert + SLA: first contact <10 minutes during working blocks.

- Failure mode: bad lead quality from top-of-funnel social traffic.
- Mitigation: enforce qualification fields (`budget_band`, `urgency`, `platform_priority`) before booking.

- Failure mode: automation brittleness from too many tools.
- Mitigation: cap v1 to max 3 automations (intake sync, booking update, payment update) and track run errors daily.

- Failure mode: over-custom CRM schema causing operational drag.
- Mitigation: lock schema to required fields only for first 14 days; add fields only when a real reporting question appears.

- Failure mode: compliance/privacy gaps in lead capture.
- Mitigation: explicit consent checkbox + privacy link on forms; avoid unnecessary PII.

**UNPROVEN:** Exact best SLA threshold (<5 min vs <10 min) for this business context.

## 4) Concrete 14-Day Execution Outline

### Days 1-2: Build core stack
- Launch Carrd page with one offer, one primary CTA (`Book Call`), one secondary CTA (`Pay Starter Deposit`).
- Set up Airtable base (`Leads`, `Activities`, `Deals`) with schema above.
- Create Stripe Payment Links for 1-2 entry offers.

### Days 3-4: Wire handoffs
- Zap 1: form submit -> upsert lead in Airtable + notify owner.
- Zap 2: Calendly booking -> update lead status to `Booked` + create activity row.
- Zap 3: Stripe payment success -> deal `payment_status=Paid`, stage `Sold`, onboarding task.

### Days 5-6: QA and dry runs
- Run 10 synthetic submissions across sources (Reddit, IG, direct) and validate every stage.
- Verify no orphan records; fix mapping errors; create fallback manual SOP for each failure point.

### Days 7-8: Go live with controlled traffic
- Push outbound/social traffic to one landing URL only.
- Daily review at end of day: leads created, booked calls, paid conversions, stuck records.

### Days 9-10: Tighten qualification
- Add/adjust 2-3 qualification questions based on low-quality lead patterns.
- Introduce simple intent scoring rules in Airtable (`budget + urgency + platform fit`).

### Days 11-12: Conversion optimization
- Improve offer clarity (deliverables, timeline, price anchor) on landing page.
- Add one proof block (sample outputs/case snippets). If no hard case data, label qualitative proof as **UNPROVEN**.

### Days 13-14: Stabilize and prep scale
- Freeze v1 process and document operating cadence (daily pipeline review, weekly source performance review).
- Define “first 10 clients” dashboard metrics: `lead->booked %`, `booked->paid %`, `time-to-first-response`, `cost per won client`.
- Decision gate: keep Option A or migrate to Option B/C only if operational pain is observed for 7 consecutive days.
