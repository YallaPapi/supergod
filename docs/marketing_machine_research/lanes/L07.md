# L07 Fulfillment Ops

## 1) Three in-lane options (SOPs + QA checkpoints + manual exception workflows)

### Option A: Lean Checklist SOP Pipeline (single operator, stage-gates)
Design:
- One canonical SOP per deliverable type (`Shorts`, `Reel`, `Reddit post`, `IG carousel`) with 7 workflow stages: intake -> brief lock -> draft -> QA-1 -> schedule -> publish -> proof-of-post.
- Mandatory checklists at `brief lock`, `QA-1`, and `proof-of-post`.
- Manual exceptions handled by a single escalation queue with response targets by severity (P1 15m, P2 4h, P3 next business day).

Evidence:
- WHO (2010-12-11): checklist use in 8-country trial reduced major complications from 11% to 7% and deaths from 1.5% to 0.8%. https://www.who.int/news/item/11-12-2010-checklist-helps-reduce-surgical-complications-deaths
- BMJ Open systematic review (2022-04-29): task-oriented checklists were more often associated with error reduction. https://pubmed.ncbi.nlm.nih.gov/35487728/
- Cureus systematic review (2024-10-06): 9/10 included studies showed lower complication rates after checklist implementation. https://pubmed.ncbi.nlm.nih.gov/39502999/

Applicability note:
- Transfer from healthcare to content ops is directionally useful but domain-shifted. **UNPROVEN**: effect size in content fulfillment will match healthcare outcomes.

### Option B: Two-Person QA Cell (creator + reviewer) with rubric scoring
Design:
- Creator submits draft against rubric (hook clarity, factual accuracy, brand voice, CTA fit, platform policy fit).
- Separate reviewer must score >= 85/100 before scheduling.
- If score 70-84: mandatory rework loop; <70: re-brief.
- Manual exceptions: reviewer can trigger `client-risk hold` when uncertainty is high.

Evidence:
- Educational Psychology Review meta-analysis (published 2025-04-21): peer-feedback with instructional support improves outcomes versus unsupported peer-feedback. https://link.springer.com/article/10.1007/s10648-025-10017-3
- Frontiers in Psychology (2023-07-14): feedback quality significantly predicted implementation quality; higher-accuracy feedback had stronger effect. https://pubmed.ncbi.nlm.nih.gov/37519361/
- Lucidpress/Marq report release (2021-08-25): 68% of respondents linked brand consistency to 10-20%+ revenue growth. https://www.prnewswire.com/news-releases/new-survey-finds-brands-struggle-to-keep-up-with-content-demand-personalized-content-could-increase-revenue-by-48-on-average-301362339.html

Applicability note:
- Academic writing evidence supports structured peer review mechanics, not agency throughput economics. **UNPROVEN**: net margin impact at small-team scale.

### Option C: Incident-Style Fulfillment Ops (severity matrix + playbooks + postmortems)
Design:
- Define fulfillment incidents: missed post window, policy rejection, wrong account publish, automation outage, client revision breach.
- Severity model (`SEV1`, `SEV2`, `SEV3`) tied to response SLA and owner.
- Exception playbooks: fallback manual posting, client comm templates, rollback/repost SOP, root-cause log + weekly postmortem.

Evidence:
- Atlassian developer guide (last updated 2023-02-23): formal incident severity levels with impact/time-based criteria. https://developer.atlassian.com/developer-guide/app-incident-severity-levels/
- Google SRE book chapter (2016): pre-defined incident management process reduced MTTR and operator stress in practice narrative. https://sre.google/sre-book/managing-incidents/
- NIST SP 800-61r2 (published 2012-08-06): standardized lifecycle (preparation, detection/analysis, containment/recovery, post-incident lessons learned). https://www.nist.gov/publications/computer-security-incident-handling-guide

Applicability note:
- Tech incident frameworks map cleanly to publishing outages and exceptions. **UNPROVEN**: full SRE-style rigor is worth overhead before 10 clients.

## 2) Lane winner with explicit score math (0-100)
Scoring model (weighted for fastest path to first 10 paying clients):
- Speed to implement (35)
- Quality protection (25)
- Client trust/retention impact (20)
- Operator overhead burden (20; higher score = lower burden)

Scores:
- Option A: speed 33/35 + quality 18/25 + trust 14/20 + overhead 16/20 = **81/100**
- Option B: speed 24/35 + quality 22/25 + trust 16/20 + overhead 10/20 = **72/100**
- Option C: speed 20/35 + quality 20/25 + trust 17/20 + overhead 8/20 = **65/100**

Winner: **Option A (81/100)**
Reason:
- Highest implementation speed with enough QA protection to prevent early-client churn.
- Minimal team-size assumptions; can run immediately with one operator and light reviewer support.
- Option B/C components should be added incrementally after first 3-5 clients.

## 3) Key failure modes + mitigations
- Failure mode: SOPs become stale after platform changes.
  - Mitigation: weekly 20-minute SOP review; version tag every checklist; block publishing when SOP version is >14 days old for that channel.
- Failure mode: QA gate becomes bottleneck and delays posting windows.
  - Mitigation: hard QA timebox (15 min per asset), fast-pass path for low-risk templates, and daily WIP cap per operator.
- Failure mode: Exceptions handled ad hoc in DMs.
  - Mitigation: force every exception into one incident log with severity, owner, ETA, and closure note.
- Failure mode: Automation failure (Geelark/IG/Reddit tools) causes missed commitments.
  - Mitigation: pre-written manual fallback SOP (mobile + desktop) and backup posting account session checks each morning.
- Failure mode: Client dissatisfaction from subjective quality disputes.
  - Mitigation: attach rubric score + checklist pass/fail snapshot to each delivery packet.

## 4) Concrete 14-day execution outline
Day 1:
- Define 4 deliverable SOP skeletons and one shared exception taxonomy (`SEV1-3`).

Day 2:
- Build QA checklist v1 (10-15 checks) and rubric v1 (100-point scale).

Day 3:
- Create intake form + brief-lock template; require approval before draft work starts.

Day 4:
- Configure fulfillment board columns: `Intake`, `Draft`, `QA`, `Scheduled`, `Published`, `Exception`.

Day 5:
- Draft manual fallback playbooks for: failed automation, policy rejection, wrong-account post.

Day 6:
- Run 5 internal dry-runs end-to-end; log misses and timing.

Day 7:
- Patch SOPs from dry-run findings; freeze `v1.0`.

Day 8:
- Start first live client batch under Option A only; track cycle time and rework count.

Day 9:
- Add proof-of-post artifacts (URL/screenshot/timestamp) as mandatory closure criteria.

Day 10:
- Review all exceptions; classify root causes into `brief`, `production`, `QA`, `publishing`, `platform`.

Day 11:
- Introduce limited Option B element: second reviewer only for high-risk assets (new niche, claims-heavy, policy-sensitive).

Day 12:
- Set SLA dashboard: on-time posting rate, first-pass QA pass rate, exception count by severity.

Day 13:
- Client-facing transparency pack: weekly ops summary with posted links, misses, fixes, next-week prevention actions.

Day 14:
- Decide go-forward: keep Option A baseline, and promote selective Option B checks if first-pass QA pass rate <85% or exception rate >10%.
